# Security Model

## Data Flow

```text
AWS Account (read-only API calls)
     |
     v
Stronghold CLI / Server (processing)
     |
     v
Local filesystem / PostgreSQL (storage)
     |
     v
Terminal / Browser (display)
```

Stronghold does not send scan data to third-party servers. There is no telemetry, no analytics, and no callback channel. The only outbound network traffic performed by Stronghold is the AWS API traffic required to execute a scan or to resolve the optional audit identity with `sts:GetCallerIdentity`.

## What Stronghold Reads

Stronghold reads AWS infrastructure metadata through read-only AWS APIs. The current AWS provider scope, service coverage, and IAM actions used by the generated policy are documented in [providers/aws.md](./providers/aws.md).

Operationally, Stronghold uses only `Describe*`, `List*`, and `Get*` style permissions plus `sts:GetCallerIdentity` for the audit trail identity lookup. It does not require or call `Create*`, `Delete*`, `Modify*`, or `Put*` APIs.

## What Stronghold Stores

CLI storage:

- `.stronghold/latest-scan.json` or `.stronghold/latest-scan.stronghold-enc`
- `.stronghold/baseline-scan.json` or `.stronghold/baseline-scan.stronghold-enc`
- generated DR plan files and runbooks when the user writes them locally
- `.stronghold/audit.jsonl`

Server storage:

- PostgreSQL metadata in the `Scan`, `ScanData`, `Report`, `DRPlan`, `PlanValidation`, `DriftEvent`, and `AuditLog` tables
- application-level scan data encryption when `STRONGHOLD_ENCRYPTION_KEY` is configured

Stored data may include:

- AWS ARNs and resource identifiers
- private and public IP addresses
- network topology and dependency information
- DR-relevant service configuration such as backup, failover, replication, and validation metadata

Stored data does not include:

- AWS credentials
- application payloads
- database contents
- S3 object bodies
- secrets retrieved from AWS services

## What Stronghold Does Not Do

- It does not execute runbook commands. It generates them; the operator executes them.
- It does not modify infrastructure.
- It does not store AWS credentials. It uses the standard AWS SDK credential chain.
- It does not send scan data anywhere other than the user-selected local filesystem or the configured PostgreSQL database.

## Threat Model

Threat 1: AWS credentials are compromised.
Mitigation: use a read-only IAM policy generated by `stronghold iam-policy`; review [providers/aws.md](./providers/aws.md) before granting access.

Threat 2: Scan results are exfiltrated from a laptop or shared filesystem.
Mitigation: use `--encrypt` for local CLI artifacts and configure `STRONGHOLD_ENCRYPTION_KEY` for server-side application encryption.

Threat 3: Raw infrastructure identifiers leak into CI logs, markdown exports, or pull request comments.
Mitigation: use `--redact` for CLI output; the server report endpoints also support `?redact=true`.

Threat 4: AGPL obligations are misunderstood during evaluation or internal deployment.
Mitigation: see [licensing-faq.md](./licensing-faq.md) for general guidance. It is not legal advice.

## Audit and Compliance

- The CLI writes an append-only JSON Lines audit trail to `.stronghold/audit.jsonl`.
- The server persists audit metadata in the `AuditLog` table and exposes `GET /api/audit`.
- The audit trail records execution metadata only. It must not include scan payloads, resource configuration dumps, credentials, or encryption passphrases.
- Stronghold is designed to process infrastructure metadata, not personal data. In the open-source self-hosted model, this generally means no DPA is required from Stronghold as a vendor because scan data stays in the user's environment.

Deployment recommendations:

- isolate the server and PostgreSQL on a private network
- terminate HTTPS at a reverse proxy
- enable PostgreSQL TLS in production
- keep containers non-root and credentials short-lived
