The Problem

Most DR plans are fiction

Your disaster recovery plan lives in a 50-page Word document. It was accurate the day it was written. That was 6 months ago. Since then: 14 infrastructure changes. 3 new services. 1 removed replica. Your plan does not know about any of it.

📄

Replica failover that no longer exists

The Plan Says

"Failover to replica"

Reality

Replica was removed 3 months ago.

The Consequence

45 min of downtime instead of 2.

☁️

Backup assumptions that drifted silently

The Plan Says

"Restore from backup"

Reality

Backup retention was reduced to 1 day.

The Consequence

72 hours of data loss instead of 1.

💥

Failover procedures that were never wired

The Plan Says

"Route53 failover"

Reality

Health check was never configured.

The Consequence

Manual DNS change at 3 AM.

What Stronghold Does

One scan. Full DR visibility.

Stronghold inspects the infrastructure you actually run, not the one your DR document still imagines. The output is versionable, reviewable, and blunt about unknowns.

🔍

Auto-Discovery

Scans 16 AWS services. Maps every dependency. Finds single points of failure before they become incident timeline trivia.

📊

DR Posture Score

Weighted score across 6 DR categories. Transparent methodology with an explicit disclaimer when the data is incomplete.

📋

DRP-as-Code

Generates a YAML DR plan you can version in Git. Includes topological recovery order, strategies, and infrastructure-backed context.

⚡

Honest RTO/RPO

Documents what AWS confirms. Says requires testing when no reliable estimate exists instead of inventing precision.

🔧

Executable Runbooks

Outputs real AWS CLI commands with your actual resource IDs. Copy-paste recovery steps, verification commands, and recovery order.

🔄

Drift Detection

Detects when DR posture degrades. Marks DRP assumptions stale when infrastructure changes invalidate the last known plan.

Quick Start

Get started in 10 seconds

Run the demo locally, then scan your own environment. The commands stay simple, the output stays local, and the first useful signal arrives quickly.

quick-start 
# Try without AWS credentials
$ npx @stronghold-dr/cli demo

# Scan your real infrastructure
$ npx @stronghold-dr/cli scan --region eu-west-1
$ npx @stronghold-dr/cli report
$ npx @stronghold-dr/cli plan runbook > recovery.yaml
🔒 Your data stays on your machine.

Read-only. Zero telemetry.

Stronghold inspects metadata using read-only APIs. It does not modify infrastructure, upload inventory, or hide uncertainty behind synthetic scores.

Architecture

Built for reliability

Stronghold is not just a CLI wrapper around cloud APIs. The repo is structured so the recovery logic stays deterministic, testable, and reusable across CLI, server, and web surfaces.

What matters operationally

TypeScript strict Core, CLI, server, and web packages all enforce strict TypeScript configs.
Ports and adapters Domain logic lives in @stronghold-dr/core; CLI and web layers orchestrate it without owning recovery rules.
Pure core package packages/core holds graph analysis, validation, DRP generation, and drift detection with minimal framework coupling.
Self-hostable The full platform runs with Docker Compose when teams need API plus web UI instead of CLI only.
AGPL-3.0 Open-source, modifiable, and self-hostable without feature gating.
